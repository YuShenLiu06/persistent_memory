# 持久上下文管理 (Persistent Context)

## 适用场景

使用此 SKILL 当：

1. **需要更新项目记忆** - 代码变更后需要同步更新 CLAUDE.md
2. **记录架构决策** - 做出重要技术决策需要创建 ADR (Architecture Decision Record)
3. **维护文档索引** - 新增或修改技术文档需要更新索引
4. **会话结束检查** - 用户结束会话前检查是否需要持久化信息
5. **用户手动触发** - 用户执行 `/update-memory` 或 `/record-refactor`

---

## 工作流程

### 流程图

```
┌─────────────────────────────────────────────────────────────┐
│                    文档更新工作流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 检测变更                                                 │
│     ├── 是否有新的架构决策？                                 │
│     ├── 是否有重要的代码变更？                               │
│     └── CLAUDE.md 状态是否需要更新？                         │
│                                                             │
│  2. 确定更新类型                                             │
│     ├── 增量更新 (推荐) - 仅修改受影响的部分                 │
│     └── 完整重建 - 仅当结构严重过时时                        │
│                                                             │
│  3. 执行更新                                                 │
│     ├── 读取现有 CLAUDE.md                                   │
│     ├── 应用增量修改                                         │
│     └── 更新时间戳                                           │
│                                                             │
│  4. 维护索引 (如需要)                                        │
│     ├── 更新 Docs/index/_tree.json                          │
│     └── 更新 Docs/index/index.md                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 增量更新策略

**优先使用增量更新**，遵循以下原则：

1. **最小改动** - 只修改必要的部分
2. **保持结构** - 不改变文档的整体格式
3. **追加优先** - 新内容追加到对应章节末尾
4. **时间戳更新** - 每次更新必须修改最后更新时间

### 更新触发条件

| 条件 | 触发动作 |
|-----|---------|
| 新增重要功能 | 更新 CLAUDE.md 的"项目结构概览" |
| 技术栈变更 | 更新"技术文档索引" |
| 架构决策 | 新增 ADR 条目 |
| 任务状态变化 | 更新"当前任务状态" |
| 用户偏好变化 | 更新"用户偏好"章节 |

## 索引维护

### 索引文件结构

项目使用双格式索引：JSON + Markdown

```
Docs/
├── index/
│   ├── _tree.json    # 结构化索引 (机器可读)
│   └── index.md      # 可读索引 (人类可读)
└── {category}/
    └── *.md
```

### _tree.json 结构规范

```json
{
  "version": "1.0",
  "updated": "YYYY-MM-DD",
  "categories": [
    {
      "name": "分类名称",
      "path": "Docs/category/",
      "description": "分类描述",
      "documents": [
        {
          "name": "文档名称",
          "file": "filename.md",
          "description": "文档描述",
          "tags": ["tag1", "tag2"]
        }
      ]
    }
  ]
}
```

### 索引更新规则

1. **新增文档** - 在对应分类的 `documents` 数组中添加条目
2. **删除文档** - 从数组中移除条目
3. **修改文档** - 仅更新描述或标签，不修改文件名
4. **新增分类** - 在 `categories` 数组中添加新分类对象

## 可用命令

### /update-memory

手动触发文档更新检查。

**使用场景**：
- 会话结束前手动保存状态
- 重要变更后立即持久化
- 定期维护项目文档

**执行流程**：
1. 读取当前 CLAUDE.md
2. 分析本次会话的变更
3. 确定需要更新的章节
4. 执行增量更新
5. 更新时间戳

### /record-refactor

记录大型重构变更。

**使用场景**：
- 大规模代码重构
- 架构重大调整
- 技术栈迁移

**执行流程**：
1. 创建新的 ADR 条目
2. 详细记录重构内容
3. 更新相关技术文档索引
4. 更新 CLAUDE.md 的项目结构概览

## 最佳实践

### DO (推荐)

- ✅ 每次重要变更后更新 CLAUDE.md
- ✅ 使用增量更新而非重写整个文件
- ✅ 为架构决策创建 ADR
- ✅ 保持文档索引与实际文件同步
- ✅ 更新时同步修改时间戳

### DON'T (避免)

- ❌ 在 CLAUDE.md 中存储敏感信息
- ❌ 频繁重写整个文档
- ❌ 忽略文档索引的维护
- ❌ 创建重复的 ADR 条目
- ❌ 忘记更新时间戳
